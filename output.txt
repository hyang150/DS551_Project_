─────────────────────────────────────────────────────  Step 2: Columnar Storage — Narrow vs Wide Scan  ──────────────────────────────────────────────────────
╭───────────────────────────────────────────────────────────────────────────────────╮
│ Key Insight: DuckDB stores data column-by-column.                                 │
│ When a query only needs 2-3 of 7 columns, DuckDB skips the rest → less I/O.       │
│ MySQL stores data row-by-row → must read all 7 columns even if only 2 are needed. │
╰───────────────────────────────────────────────────────────────────────────────────╯

Query A: Narrow Scan — reads only Close + Volume (2/7 cols)
SELECT COUNT(*), ROUND(AVG(Close), 4), SUM(Volume)
    FROM stock_data

Query B: Wide Scan — reads all 7 columns (SELECT *)
SELECT * FROM stock_data ORDER BY Date DESC LIMIT 5000
                   Columnar Storage: Narrow vs Wide                   
╭───────────────────┬───────────┬─────────────┬────────────┬─────────╮
│ Query             │ Cols Read │ DuckDB (ms) │ MySQL (ms) │ Speedup │
├───────────────────┼───────────┼─────────────┼────────────┼─────────┤
│ Narrow (AVG, SUM) │ 2 / 7     │         1.3 │       10.6 │    8.0x │
├───────────────────┼───────────┼─────────────┼────────────┼─────────┤
│ Wide (SELECT *)   │ 7 / 7     │         6.8 │       62.5 │    9.1x │
╰───────────────────┴───────────┴─────────────┴────────────┴─────────╯

Takeaway: DuckDB's narrow scan is much faster because columnar storage
only reads the needed columns. The advantage shrinks on SELECT * (wide scan)
because all columns must be read and reconstructed into rows.
